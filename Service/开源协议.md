# 开源协议

<https://choosealicense.com/licenses/>

## 极度宽松型

### BSL-1.0

Boost Software License 1.0

一种极度宽松、简单且对商业极其友好的开源协议

- 二进制分发不强制要求署名

### MIT

MIT License
最简洁、最流行的协议。只需保留版权声明，怎么用都行。

- 想让更多人使用： 选 MIT
- 无论源码还是二进制分发，都必须保留署名。

### Apache 2.0

Apache License 2.0
比 MIT 多了专利授权条款（保护用户不被贡献者起诉专利侵权），更适合大企业。

- 大企业合作、防专利纠纷： 选 Apache 2.0。
- 可闭源

### BSD

类似于 MIT，但通常分为两句版和三句版（限制使用作者名义做宣传）。

- 可闭源
- 无论源码还是二进制分发，都必须保留署名。

## 弱保护型

### LGPL

GNU LGPLv3
通常用于库文件。如果你只是链接（Link）该库，你的主程序可以闭源；如果你改了库本身的代码，修改部分必须开源。

- 部分闭源 仅对开源库本身的修改需开源

### MPL (Mozilla Public License)

以“文件”为界限。修改该开源文件需开源，但你新创建的文件可以闭源。

- 部分闭源

## 强保护型 / 传染型（Strong Copyleft）

### GPL (v2/v3)

GNU GPLv3
最著名的保护协议。只要你的代码里用了 GPL 的代码，你的整个项目都必须开源。

- 防止代码被大厂拿去闭源赚钱： 选 GPLv3。
- 不可闭源，具有传染性，关联项目必须开源。

### AGPL (v3)

GNU AGPLv3
最严格。在 GPL 基础上增加了“网络交互”条款。即便不卖软件，只在云端运行提供服务（SaaS），也必须公开源码。

- 做云服务（SaaS）且不想被白嫖： 选 AGPLv3。
- 不可闭源，具有传染性，关联项目必须开源。
